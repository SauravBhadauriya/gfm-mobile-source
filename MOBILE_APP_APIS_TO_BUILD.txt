CATEGORIES (1 API)

GET /categories
Query Parameters: page, limit
Returns: Array of category objects with id, name, icon, image

Explanation: The mobile app displays categories on the home screen and category detail pages. Currently, the app tries to fetch categories from the admin endpoint which requires authentication. This causes issues for users who are not logged in or have expired tokens. We need a public categories endpoint that anyone can access without authentication. This allows users to browse categories and competitions by category before logging in, improving user experience and reducing authentication-related errors.

COMPETITIONS (12 APIs)

GET /competitions
Query Parameters: status, category, page, limit, search
Returns: List of competitions with all details

Explanation: This is the main endpoint for fetching competitions in the mobile app. Users need to see all available competitions on the competitions listing screen. The app filters competitions by status (upcoming, live, past) and allows searching by category or keyword. This endpoint must be public so users can browse competitions without logging in. Competitions created in the admin panel should automatically appear here when their status changes to live.

GET /competitions/upcoming
Query Parameters: page, limit
Returns: List of upcoming competitions

Explanation: The app has a dedicated screen for upcoming competitions where users can see competitions that are about to start. Users can join these competitions, but they must complete KYC verification first. This endpoint helps users discover new competitions they can participate in.

GET /competitions/live
Query Parameters: page, limit
Returns: List of live competitions

Explanation: Live competitions are currently active and accepting entries. Users can view these competitions, vote on entries, and see leaderboards. Unlike upcoming competitions, users do not need KYC verification to view or vote in live competitions. This endpoint powers the live competitions screen in the app.

GET /competitions/past
Query Parameters: page, limit
Returns: List of past competitions

Explanation: Past competitions show historical competitions that have ended. Users can view winners, see final leaderboards, and check their own participation history. This endpoint is used in the past competitions screen and competition history feature.

GET /competitions/{id}
Returns: Single competition with full details

Explanation: When a user taps on a competition card, they need to see complete details including rules, prizes, dates, entry fee, participants count, and other information. This endpoint provides all the data needed to display the competition detail screen. It is used in upcoming, live, and past competition detail screens.

POST /competitions/{id}/join
Body: userId
Returns: Join status and confirmation

Explanation: When a user wants to participate in an upcoming competition, they must join it first. This endpoint creates a participation record and validates that the user has completed KYC verification. If KYC is not complete, the join request is rejected. After joining, users can upload their competition entries.

GET /competitions/{id}/participants
Query Parameters: page, limit
Returns: List of participants with their details

Explanation: Users want to see who else is participating in a competition. This endpoint lists all participants, showing their profile information, entry count, and current ranking. It helps users understand the competition level and see their competitors.

GET /competitions/{id}/leaderboard
Query Parameters: page, limit
Returns: Leaderboard with rankings and vote counts

Explanation: During live competitions, users can see real-time leaderboards showing who is winning based on votes. This endpoint provides ranked list of participants with their vote counts, helping users understand the competition standings and make informed voting decisions.

GET /competitions/{id}/winners
Returns: List of winners with their ranks and prizes

Explanation: After a competition ends and winners are declared by the admin, users need to see who won. This endpoint shows the winners with their ranks, prize amounts, and coins awarded. It is displayed on past competition detail screens.

POST /competitions/{id}/entries
Body: FormData with video, thumbnail, caption, category
Returns: Entry created confirmation

Explanation: After joining a competition, users upload their video entries. This endpoint handles the video upload, creates an entry record linked to the competition, and processes the video for moderation. The uploaded entry becomes visible in the competition's entry list and can receive votes.

GET /competitions/{id}/entries
Query Parameters: userId, page, limit
Returns: List of competition entries

Explanation: Users need to see all entries submitted for a competition. This endpoint lists all entries with video thumbnails, captions, vote counts, and participant information. Users can browse entries, watch videos, and vote on them. Filtering by userId allows users to see only their own entries.

POST /competitions/{id}/entries/{entryId}/vote
Body: userId
Returns: Vote confirmation and updated vote count

Explanation: Users can vote on competition entries during live competitions. This endpoint records the vote, increments the vote count for the entry, and tracks voting history to prevent duplicate votes. Users can vote once per day per competition. The vote count is used to determine rankings in the leaderboard. This is essential for the competition voting system.

==========================================
SPECIAL NOTES
==========================================

1. Categories API must be public endpoint /categories, not /admin/categories. Currently causes authentication errors.

2. Competition flow: Admin creates competition via admin panel, app fetches via GET /competitions (public), user joins via POST /competitions/{id}/join, user uploads entry via POST /competitions/{id}/entries, admin declares winners via admin panel, backend auto-creates earnings records.

3. Reel moderation: User uploads via POST /reels/upload, backend AI/ML processes and auto-approves/rejects, approved reels visible, rejected reels hidden.

4. Earnings auto-update: Competition win creates earnings automatically, tip received creates earnings automatically, both visible in app and admin panel.

5. KYC auto-verification: User completes all steps, frontend checks completion, backend updates KYC status to completed.

6. Wallet API GET /user/wallet is already implemented and returns balance, totalEarnings, totalTips.

7. User profile must include bio and three-word description fields. Refer to the "Own Profile" screen in Figma for the exact format.

8. Block functionality: When a user blocks another user (either from profile or chat), the blocked user cannot see the blocker's content, send messages, or interact in any way. Blocked users are hidden from the blocker's feed and search results.

9. Notification types: The system must support all notification types including COMPETITION_UPDATE, WINNER_ANNOUNCEMENT, NEW_FOLLOWER, TIP_RECEIVED, COMMENT, LIKE, RANK_BOOST, DAILY_CHALLENGE, RAP_BATTLE_INVITE, UPCOMING_EVENT, PROFILE_UPDATE, COMPETITION_RESULT, PAYMENT_SUCCESS, and SYSTEM_MESSAGE.

10. Community and Referral features are NOT included in this API list as they are not implemented in the mobile app.
